# Marker 产品打磨 TODO

> 目标：先在 `iOS` 上验证 `快记`、`低负担`、`可信`、`可回看`、`可扩展` 的核心体验；`Android` 暂时保持当前基线，等 iOS 试点稳定后再对齐。
> 产品原则与品牌调性见 `marker-tone.md`，本文件只保留待执行事项。


## P0 iOS 试点方向

- [ ] 明确 iOS 首轮试点范围：习惯、用药、经期、自定义四类 tracker 的最低可用体验
- [ ] 用 iOS 验证不同 tracker 类型的创建、记录、回看和统计表达，再决定跨端契约是否需要调整
- [ ] 保持 Android 暂不追新功能，只在共享契约稳定后再进行平台对齐

## P0 iOS 记录编辑闭环

- [ ] 支持选择历史 `dayKey` 进行补记
- [ ] 支持改记已有记录，包括 payload、备注和记录时间语义
- [ ] 支持删除记录，并提供清楚的确认与恢复反馈
- [ ] 清楚展示记录时间、逻辑日归属和记录时区来源
- [ ] 统一 iOS 侧 `countsAsCompletion` 口径，避免用药跳过、备注记录等场景被误算成完成

## P0 iOS Today 首页重构

- [ ] 重做 `Today` 首页结构，让用户一眼看懂 `今天该记什么 / 已经记了什么 / 还缺什么`
- [ ] 区分不同 tracker 类型的入口和交互密度，不把所有场景都按“习惯打卡”处理
- [ ] 为习惯、用药、经期、自定义记录提供更贴近场景的行内状态
- [ ] 空状态、保存成功、异常失败都给出温和且明确的反馈

## P1 iOS 创建与管理体验

- [ ] 模板化创建追踪项，至少支持 `习惯`、`用药`、`经期`、`自定义`
- [ ] 习惯型记录支持 `完成 + 备注`
- [ ] 用药型记录支持 `已服用 / 已跳过 / 剂量 / 单位 / 备注`
- [ ] 经期型记录支持 `流量 / 症状 / 备注`
- [ ] 自定义记录先支持 `备注型`，后续扩展为 `数值型 / 状态型`
- [ ] 归档入口、低频配置、危险操作集中放在 `Settings`

## P1 iOS History 与 Statistics

- [ ] 历史页支持 `最近记录时间线`、`按天详情`、`类型筛选`
- [ ] 历史明细直接展示 payload 摘要，而不是只显示“记录了一次”
- [ ] 统计页按类型使用不同统计口径，而不是统一看成完成率
- [ ] 习惯场景强调坚持和稳定反馈
- [ ] 用药场景强调准确、可信、低出错
- [ ] 经期场景强调隐私、细腻、可回看
- [ ] 自定义场景强调灵活和低门槛

## P1 iOS 信任感

- [ ] 清楚说明当前是 `本地优先`
- [ ] 清楚说明当前是否有 `同步 / 备份`
- [ ] 重要操作支持撤销或恢复
- [ ] 异常场景不能静默失败，要给出可理解反馈
- [ ] 做基础导出入口或备份说明，避免用户误以为已有云同步

## P2 iOS 高级增强

- [ ] 追踪项支持 `类别 / 分组`
- [ ] `Today` 支持按类别筛选或折叠
- [ ] 增加更通用的 `payload` 类型：`numeric / duration / range`
- [ ] 支持 `纯事件型 tracker`，不强依赖 `schedule`
- [ ] 支持完整 `导出 / 备份 / 恢复`
- [ ] 支持提醒调度与 `TrackerReminder` 平台落地
- [ ] 支持默认首页 tab 的用户可配置化
- [ ] 支持 `小组件 / 快捷指令`
- [ ] 支持更智能的模板和默认配置

## P3 Android 对齐

- [ ] Android 补齐 `TrackingPayload`，对齐 iOS 试点稳定后的记录语义
- [ ] Android Room schema 增加 payload 存储与迁移，保证历史记录可继续读取
- [ ] Android Today / History 展示 payload 摘要，而不是只显示“记录了一次”
- [ ] Android 对齐 iOS 的补记、改记、删除、Today 首页和类型化统计体验
