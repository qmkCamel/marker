# habit-schedule-rules Specification

## Purpose
定义 MVP 阶段支持的追踪项频率类型，以及系统如何判断某个追踪项在某个逻辑日是否应完成，以便 Today、历史和统计能力共享同一套规则。

## Requirements
### Requirement: MVP 支持一组明确的频率类型
系统必须（MUST）支持以下 MVP 频率类型：
- `daily`
- `weeklyOnDays`，包含一个或多个明确星期
- `weeklyQuota`，包含目标完成次数，且周边界由用户偏好决定

系统必须将月度规则、固定间隔规则和自然语言规则视为本次变更的范围外内容。

#### Scenario: 产品或工程决策引用支持的频率
- **WHEN** 开发者查看 MVP 的频率规则
- **THEN** 支持的频率类型会被明确列出
- **AND** 暂不支持的频率家族会被清楚标记为延期内容

### Requirement: 频率计算定义追踪项在某个逻辑日是否应完成
系统必须（MUST）为每一种支持的频率类型定义“某个逻辑日是否应完成”的判断规则。

这些判断规则必须满足：
- `daily`：只要追踪项处于激活状态，每一个逻辑日都应完成
- `weeklyOnDays`：只有当逻辑日落在选中的星期上时才应完成
- `weeklyQuota`：在当前周的记录次数达到目标前，该周内的每个逻辑日都应继续视为应完成

#### Scenario: 仅在选中的星期显示该追踪项
- **WHEN** 某个追踪项使用 `weeklyOnDays`，并选择周一、周三和周五
- **THEN** 该追踪项会在逻辑上的周一、周三和周五被视为应完成
- **AND** 在其他逻辑星期中不会被视为应完成

#### Scenario: 周配额在达到目标前持续应完成
- **WHEN** 某个追踪项使用 `weeklyQuota`，目标为每周完成三次
- **AND** 当前周截至目前只有两次有效记录
- **THEN** 该追踪项在当前逻辑日仍然应完成
- **AND** 一旦当前周记录了第三次记录，该追踪项在本周剩余时间内不再视为应完成

### Requirement: 提醒时间不重定义应完成逻辑
系统必须（MUST）将提醒配置视为通知关注点，而不是会改变追踪项在某个逻辑日是否应完成的规则。

#### Scenario: 结合频率查看提醒配置
- **WHEN** 开发者同时查看某个追踪项的频率和提醒设置
- **THEN** 频率仍然是判断该追踪项是否应完成的真相源
- **AND** 提醒设置只影响通知投递，而不改变应完成判断
