## Context

仓库已经有 iOS 应用壳子、共享领域契约和 AI/OpenSpec 规则，但产品仍然只是一个占位应用。要进入可用状态，必须让 iOS 成为本地优先的真实习惯追踪应用，至少完成本地持久化、习惯管理、Today 打卡、历史和统计这条闭环。

约束：
- 当前只实现 iOS，本地优先，不做账号和云同步。
- 保持既有的 `MarkerDomain -> MarkerData -> MarkerApp` 边界。
- 本地存储采用 SQLite，并与 `dayKey` 语义保持一致。
- 本轮不实现系统提醒调度和跨端同步。

## Goals / Non-Goals

**Goals:**
- 引入稳定的本地持久化和仓储实现。
- 让用户可以创建、编辑、归档习惯并在 Today 完成打卡。
- 提供可用的历史浏览与基础统计反馈。
- 用最小但完整的设置能力承接周起始日和统计偏好。

**Non-Goals:**
- 不实现登录、同步、云端 API。
- 不实现 Android/Web/Backend 真正业务。
- 不实现系统级通知提醒调度。
- 不追求首轮就做到复杂月历交互或高阶分析图表。

## Decisions

### 1. 使用 GRDB + SQLite 作为本地持久化基础

本轮将 `MarkerData` 建立在 GRDB 之上，使领域契约能够稳定映射到 SQLite，并为后续迁移与同步预留足够控制力。

备选方案：
- SwiftData：实现速度更快，但对 schema、迁移和同步友好度不如 GRDB，放弃。

### 2. 让 MarkerApp 通过单一应用模型协调页面状态

本轮会用一个应用级模型装配仓储、刷新 Today、历史和统计数据，避免在功能刚起步时就为每个页面过度拆分基础设施。

备选方案：
- 每个页面单独拉取数据：容易造成重复查询和状态不一致，放弃。

### 3. 历史与统计保持“足够可用”而不是一步到位

为了尽快把应用做成可用产品，历史视图先提供按逻辑日浏览和明细，统计先提供摘要卡片和基础明细，不在本轮实现复杂月历交互与高级图表。

备选方案：
- 先做完整月历与高阶分析：投入大、会拖慢闭环落地，放弃。

### 4. 设置页只承接低频配置与管理入口

设置页会负责周起始日、统计窗口、归档入口和基础应用信息，高频行为仍然聚焦在 Today。

## Risks / Trade-offs

- [首轮应用模型可能偏集中] -> Mitigation: 先保证完整闭环可用，后续再按功能拆细。
- [未做系统提醒可能与增强版 MVP 有差距] -> Mitigation: 保留 Reminder 领域契约，下一轮再接入系统调度。
- [历史视图不做复杂月历可能与 HabitKit 体验存在差距] -> Mitigation: 先提供稳定可用的历史浏览，再迭代视觉和交互。

## Migration Plan

1. 引入 GRDB 并完成本地 schema 与迁移。
2. 为 `MarkerData` 增加仓储实现和测试。
3. 用真实数据驱动 iOS 页面，替换占位视图。
4. 跑通构建与测试，确认应用可正常运行。

## Open Questions
